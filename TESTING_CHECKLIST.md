## قائمة اختبار نظام الإحصائيات الشامل

### ✅ الاختبارات الأساسية

#### 1. التحقق من البيانات الأساسية
- [ ] التحقق من تشغيل جميع migrations بنجاح
```bash
php artisan migrate:status | grep 2026_03
```

- [ ] التحقق من وجود الأدوار العربية
```bash
php artisan tinker -c "Spatie\Permission\Models\Role::where('name', 'رئيس قسم')->first()"
```

- [ ] التحقق من وجود أعمدة جديدة في جداول قاعدة البيانات
```
payrolls: user_id ✅
payrolls: created_by_department_id ✅
users: department_id ✅
departments: parent_id ✅
```

---

#### 2. اختبار Dashboard UI

**الخطوات:**
1. أنشئ مستخدم اختبار
2. سجل الدخول بهذا المستخدم
3. ادخل إلى `/dashboard`

**التحقق:**
- [ ] يظهر تابان: "احصائياتي" و "احصائيات الفريق"
- [ ] التاب "احصائياتي" نشط افتراضياً
- [ ] يمكن النقر على التاب الثاني
- [ ] الفلاترات موجودة (السنة، الشهر، اليوم)

---

#### 3. اختبار فلاترة التاريخ (أهم اختبار)

**الخطوات:**
1. في تاب "احصائياتي" اختر سنة (مثل 2026)
2. لاحظ الأشهر المتاحة في القائمة الثانية
3. اختر شهر (مثل مارس)
4. لاحظ الأيام المتاحة في القائمة الثالثة
5. اختر يوم (مثل 5)

**النتيجة المتوقعة:**
- [ ] عند اختيار السنة: الأشهر تتحدث لكن لا تعود إلى الصفر
- [ ] عند اختيار الشهر: الأيام تتحدث لكن الشهر لا يتغير
- [ ] عند اختيار اليوم: البيانات تحدثت بشكل صحيح
- [ ] **أهم شيء**: الأرقام في البطاقات تختلف (ليست مكررة)

**المشكلة القديمة التي تم إصلاحها:**
```
❌ قبل: اليوم 5 → تصبح جميع الأرقام: الكل=5, السنة=5, الشهر=5, اليوم=5
✅ بعد: اليوم 5 → كل بطاقة تعرض الرقم الصحيح الخاص بها
```

---

#### 4. اختبار API Endpoints

**في Postman أو curl:**

```bash
# 1. الحصول على السنوات
GET http://localhost/api/user-years
Headers: Authorization: Bearer {TOKEN}
         X-Requested-With: XMLHttpRequest

# 2. الحصول على الأشهر
GET http://localhost/api/user-months?year=2026
Headers: نفس الأعلى

# 3. الحصول على الأيام  
GET http://localhost/api/user-days?year=2026&month=3
Headers: نفس الأعلى

# 4. الحصول على الإحصائيات
GET http://localhost/api/user-stats?year=2026&month=3&day=5
Headers: نفس الأعلى

✅ الاستجابة المتوقعة:
{
  "total_payrolls": 3,
  "total_amount": 15000
}
```

#### 5. اختبار Role-Based Filtering

**السيناريو 1: موظف عادي (data-entry)**
- [ ] في "احصائياتي": يرى الكشوفات التي أنشأها فقط
- [ ] في "احصائيات الفريق": لا يرى الخيار (لا دور إداري)

**السيناريو 2: مسؤول وحدة**
- [ ] في "احصائياتي": يرى الكشوفات التي أنشأها
- [ ] في "احصائيات الفريق": يرى كشوفات وحدته فقط

**السيناريو 3: مسؤول شعبة**
- [ ] في "احصائياتي": يرى الكشوفات التي أنشأها
- [ ] في "احصائيات الفريق": يرى 
  - كشوفات شعبته المباشرة
  - كشوفات جميع الوحدات التابعة للشعبة

**السيناريو 4: رئيس قسم / admin**
- [ ] في "احصائياتي": يرى الكشوفات التي أنشأها
- [ ] في "احصائيات الفريق": يرى **جميع الكشوفات** (لا توجد فلاترة)

#### 6. اختبار Payment Tracking

**تتبع من أنشأ الكشف:**

```php
// في tinker بعد إنشاء كشف جديد:
$payroll = Payroll::latest()->first();
echo $payroll->user_id;  // يجب أن تكون معرف المستخدم الحالي
echo $payroll->created_by_department_id;  // يجب أن يكون قسم المستخدم
```

- [ ] كل كشف جديد يحتوي على user_id صحيح
- [ ] كل كشف جديد يحتوي على created_by_department_id صحيح

### 🐛 معالجة الأخطاء

#### إذا حصلت على خطأ 401 (Unauthorized)
```
السبب: Missing authentication token
الحل: 
- تأكد من وجود Authorization header
- تأكد من استخدام token صحيح
```

#### إذا لم تظهر التابين
```
السبب: قد تكون JavaScript لم تحمل بشكل صحيح
الحل:
- F12 → Console وابحث عن أخطاء
- تحديث الصفحة Ctrl+Shift+Delete
- تحقق من تحميل آخر نسخة من CSS/JS
```

#### إذا كانت البيانات فارغة
```
السبب: قد لا يكون لديك كشوفات للمستخدم
الحل:
- إنشاء كشف جديد عبر واجهة النظام
- التأكد من أن user_id و department_id متوافقة
```
